## 1. Executive Summary

**Balance** is a local-first, "learn-by-doing" algebra web app modeled on Brilliant.org. It teaches
a single coherent path — **Algebra Foundations** — through interactive, typed lesson steps with
instant, authored feedback. The flagship interaction is a manipulable **balance scale**.

**Overall health: strong for a Phase-1 MVP.** Build, lint, and the test suite are all green, the
architecture cleanly separates the content model, the answer-checking engine, and the persistence
backend, and the local learning loop is fully playable end-to-end across all six lessons.

**The single biggest gap is the hosted backend.** Everything runs against a browser-local
`LocalBackend`; Firebase is scaffolded but not wired into the runtime, so several P0 PRD
requirements (cross-device sync, real auth, security rules in effect, public deploy, concurrency)
are deferred. The **second** notable gap is documentation drift: lessons 4–6 are described as
unbuilt "shells" but are in fact fully authored, and the docs describe an "EWMA" mastery model that
the code does not implement.

## 2. What the Project Is (Context)

- **Subject:** Algebra only — a deliberate depth-over-breadth bet (see PRD "Non-Goals").
- **Persona:** "Maya," a new-to-algebra 8th grader, mobile-first.
- **Thesis:** Every lesson is a short sequence of *typed, interactive steps* rendered from a data
  model, with sub-100 ms client-side answer checking and specific authored feedback for both right
  and common wrong answers — never a bare red ✗.
- **Course path (all six lessons authored):**
  1. Balancing Equations (flagship, balance scale)
  2. One-Step Equations
  3. Two-Step Equations
  4. Like Terms & Variables on Both Sides
  5. Coordinate Plane
  6. Graphing Lines

## 3. Architecture & Tech Stack

**Stack:** React 19.2 + TypeScript ~6.0 + Vite 8, ESLint 10 / typescript-eslint 8, Node's built-in
test runner. Firebase SDK (v12.15) is a dependency but is currently scaffolding only. No CSS
framework (hand-written `App.css`, ~1.5k lines).

**Module layout (`src/`):**

| File | Lines | Responsibility |
| --- | --- | --- |
| `domain.ts` | 1680 | Content model types + **all authored lesson content** + course/skill catalogs |
| `App.tsx` | 1805 | Entire React UI: all screens, step renderers, balance drag interaction |
| `engine.ts` | 563 | Pure answer-checking + progress/score/mastery/recommendation logic |
| `backend.ts` | 487 | `Backend` contract + `LocalBackend` (localStorage/sessionStorage) + provider factory |
| `firebaseConfigCore.ts` / `firebaseConfig.ts` / `firebaseServices.ts` | 76 / 21 / 29 | Env-driven Firebase config + (unused-at-runtime) SDK init |
| `App.css` | 1483 | All styling, hand-written |
| `main.tsx` | 10 | React entry |

**Content model:** A `Lesson` is an ordered array of tagged-union `LessonStep`s
(`concept | mcq | input | balance | operation-choice | sequence`). The six lessons contain ≈51
authored steps: 15 concept, 15 operation-choice, 9 sequence, 7 input, **4 balance**, and **1 MCQ**.

**Design strengths:**

- **Clean separation of concerns.** The content model (`domain.ts`), the pure checking/engine logic
  (`engine.ts`), and persistence (`backend.ts`) are decoupled. The engine functions are pure and
  well unit-tested.
- **Backend behind an app-owned interface.** React talks only to the `Backend` contract; provider
  SDKs are meant to live in adapters (`BACKEND_ADAPTERS.md`). This makes the eventual Firebase swap
  a contained change.
- **Fails closed.** Selecting `VITE_BACKEND_PROVIDER=firebase` before the adapter exists throws a
  clear startup error (`createBackend`) rather than silently falling back to local demo storage —
  good safety posture, surfaced to the user via `BackendConfigurationError`.
- **Defensive persistence.** `LocalBackend` normalizes/validates everything it reads from storage
  (`normalizeDatabase`, per-record validators), recovers from corrupt JSON, scopes data access to
  the active session user, and never persists passwords. This is well covered by tests.

## 6. Current Issues (concrete, present today)

1. **Documentation drift: lessons 4–6 are described as "shells" but are fully built.**
   `README.md` (lines ~3, 29–30, 92) and `PHASE1_QA_CHECKLIST.md` (lines ~53–54, 83) describe Like
   Terms, Coordinate Plane, and Graphing Lines as "lightweight content shells" and list "fill the
   Lesson 4–6 shells" as a *next priority*. In reality all six lessons in `domain.ts` have complete
   interactive step arrays and pass the catalog test. The docs understate the project's actual
   progress and should be updated.

2. **Mastery model mismatch (docs vs. implementation).** `BACKEND_ADAPTERS.md` (line 14) says
   "per-skill EWMA score" and the PRD suggests an EWMA, but `backend.ts` `updateSkillMastery` stores
   a plain cumulative ratio `score = correct / attempts` (line ~387). Consequences: early attempts
   count forever (no recency weighting/decay), so a learner who improves stays penalized, and the
   model can't support the Phase-3 decay/spaced-repetition plan without change.

3. **Skill attribution is coarse.** On each assessed step, `App.tsx` (lines ~147–148) updates
   mastery for **every** `skillId` of the active lesson with the same correct/incorrect signal,
   regardless of which skill the step actually exercises. For multi-skill lessons (e.g. Balancing
   Equations → `equality` + `inverse-operations`; Like Terms → `like-terms` +
   `variables-on-both-sides`) the two skills' scores are always identical, making per-skill mastery
   less meaningful than it appears.

4. **MCQ logic lives in the view and is untested.** Unlike every other step type, there is no
   `checkMcqStep` in `engine.ts`; the MCQ correctness + escalation logic is duplicated inline in
   `MultipleChoiceStep` in `App.tsx` (lines ~778, 797–814) and has no unit test. This is an
   inconsistency and a small correctness risk (the escalation rules are re-implemented by hand).

5. **Dead `status` field + dead "coming-soon" path.** `CourseLessonNode.status`
   (`'available' | 'locked' | 'coming-soon'`) in `domain.ts` (line ~161, set per-lesson in
   `algebraCourse`) is never read by the UI; `App.tsx` computes status itself and treats a lesson as
   "coming soon" only when `steps.length === 0` (line ~518), which never happens now. The
   `.coming-soon` styling and the `comingSoon` branch are effectively dead code.

6. **`App.tsx` reads persistence on every render.** Functions like `getProgressByLesson`,
   `backend.mastery.getUserMastery`, and `backend.attempts.getAttempts` run during render (lines
   ~100–101, 252) and each performs a `localStorage` read + `JSON.parse` + full normalization (and
   `getProgressByLesson` loops over all lessons). Harmless at MVP scale, but wasteful and a smell
   that will matter when the backend becomes async.

7. **Unused runtime dependency footprint.** `firebase` is installed and `firebaseServices.ts`
   exists, but nothing imports it at runtime yet. It's correctly tree-shaken from the bundle, but
   it's worth being explicit that the dependency is presently inert.

8. **No CI and no React-layer tests.** There is no `.github/` workflow, so lint/test/build are not
   enforced on push, and the React UI (`App.tsx`, including the balance drag) has no automated
   coverage — only the manual QA checklist.

## 7. What Needs To Be Done (gaps to "complete")

**To satisfy the remaining P0 PRD requirements (highest impact):**

- **Wire a real hosted backend (Firebase).** Implement `FirebaseBackend` behind the existing
  `Backend` interface using `firebaseServices.ts`; this unlocks R1/R2 (real auth + durable login),
  R5 (password reset), R25/N5 (cross-device resume), N9 (concurrency), N11/N12 (rules actually
  enforced), and N13 (public deploy). `BACKEND_ADAPTERS.md` already lays out the recommended order.
- **Make the `Backend` contract async first.** The contract is synchronous because `LocalBackend`
  uses localStorage; Firestore is async. Convert repository methods to promises and add
  loading/error states in the React flows *before* swapping providers (the docs flag this).
- **Deploy publicly** (Firebase Hosting is already configured in `firebase.json`) and run a basic
  concurrent-user smoke test.

**To deliver the interactions the PRD actually calls for (lessons 5–6 currently fall short):**

- Build the **interactive coordinate grid** (R15) for Coordinate Plane (tap/drag to plot).
- Build the **slope/intercept sliders + live line render** (R16/R19/R20) for Graphing Lines.
- Optionally build **draggable term tiles across `=`** (R14) for Like Terms.
  These three lessons are currently text/tap puzzles without the responsive visual that is the
  product's core differentiator.

**Engineering hygiene:**

- Add a `checkMcqStep` to `engine.ts` and unit-test it (close the testing gap).
- Update `README.md`, `PHASE1_QA_CHECKLIST.md`, and `BACKEND_ADAPTERS.md` to reflect that lessons
  4–6 are authored and that mastery is a cumulative ratio (not EWMA).
- Remove or actually use the dead `CourseLessonNode.status` field and `coming-soon` styling.
- Add CI (lint + test + build on every push).

## 8. Potential / Future Issues (risks to watch)

- **Async migration blast radius.** Because the entire UI consumes a synchronous backend and reads it
  during render, the Firebase swap touches many call sites. Doing the async refactor while the local
  backend is still the only implementation (so tests stay green) reduces risk.
- **Cross-device sync/conflict semantics are unspecified.** Once two devices can write progress,
  last-write-wins on whole `LessonProgress`/`SkillMastery` documents could clobber concurrent
  updates. Decide on merge vs. last-write semantics before launch (the BrainLift flags this same
  question about Brilliant itself).
- **Mobile drag relies on `document.elementFromPoint`.** The balance drag detects the drop pan via
  point hit-testing. This is generally fine with Pointer Events + pointer capture, but is worth
  explicit testing on real iOS/Android Safari/Chrome and with assistive tech; the keyboard/tap
  fallback mitigates the worst case.
- **Single-file growth.** `App.tsx` (~1.8k lines) and `domain.ts` (~1.7k of mostly inline content)
  are becoming monolithic. As lessons grow, this hurts navigability, review, and reuse.
- **Content scaling strategy.** All content is bundled in `domain.ts`. The PRD anticipates moving to
  fetch-and-cache from Firestore as the library grows; the bundle (and TS compile time) will grow
  with each authored lesson until then.
- **COPPA / age gating is unresolved** (PRD open question) for a teen audience — must be settled
  before any real account collection.
- **No error boundary / runtime monitoring.** A render error currently has no React error boundary;
  Crashlytics/Sentry-style monitoring (PRD measurement section) is not present.
