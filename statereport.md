coherent path — **Algebra Foundations** — through interactive, typed lesson steps with instant,
authored feedback. The flagship interaction is a manipulable **balance scale**, joined by an

## 2. What the Project Is (Context)

- **Subject:** Algebra only — a deliberate depth-over-breadth bet (PRD "Non-Goals").
- **Persona:** "Maya," a new-to-algebra 8th grader, mobile-first.
- **Thesis:** Every lesson is a short sequence of *typed, interactive steps* rendered from a data
  model, with sub-100 ms client-side answer checking and specific authored feedback for both right
  and common wrong answers — never a bare red ✗.
- **Course path — a branching DAG** (not a straight line): Balancing Equations → One-Step →
  Two-Step, then Two-Step **diverges** into both **Like Terms** and **Coordinate Plane**, which
  **merge** as prerequisites of **Graphing Lines**. `prerequisites` on each lesson encode the graph;
  `lessonOrder` keeps a valid topological order for recommendations.

**Content model (`content/types.ts`):** A `Lesson` is an ordered array of tagged-union `LessonStep`s.
Step types number **ten**: `concept`, `mcq`, `input`, `operation-choice`, `sequence`, `balance`,
plus four hands-on types — **`manipulative`** (equal-groups / collect / build-product puzzles),
**`plot`** (interactive coordinate grid), **`slider`** (slope/intercept → live line), and
**`dragTerms`** (term-tile sorting). The engine exposes **eight** pure `check*Step` functions; of the
nine *assessed* step types, only **MCQ lacks an engine checker** (it is checked in the view — §6.6);
`concept` needs none. Each lesson file is 192–361 lines, fully authored — **none are "shells"**:

- **Balancing Equations** (277 lines), **One-Step** (282), **Two-Step** (361), **Like Terms &
  Variables Both Sides** (321), **Coordinate Plane** (192), **Graphing Lines** (218).

**Equation rendering (new in `9f8ed3c`):** `equationToLatex` converts the app's plain authoring
notation (e.g. `"3x - 5 = 19"`, `"x / 6 = 2"`, `"x - 5 = 9 -> x = 9"`) into LaTeX with conservative,
fail-soft transforms (multi-letter prose → `\text{}`, `->` → `\rightarrow`, `a/b` → `\frac{a}{b}`,
`*` → `\cdot`). `MathText` renders it via `katex.render` into a ref (no `dangerouslySetInnerHTML`),
with `throwOnError:false`, an `aria-hidden` visual layer, and a visually-hidden plain-text label for
screen readers — a thoughtful accessibility design. KaTeX CSS is imported once in `main.tsx`.

**Design strengths:**

- **Clean separation of concerns.** Content (`content/`), pure logic (`engine.ts`), and persistence
  (`backend.ts` / `firebaseBackend.ts`) are decoupled; the engine functions are pure and well tested.
  Lessons are one file each, which supports parallel authoring.
- **Provider-neutral backend behind one contract.** React talks only to `Backend`; both `LocalBackend`
  and `FirebaseBackend` implement it. The contract is **async-capable** (`MaybePromise`), and the UI
  loads learner data into React state and handles loading/error states (no reads during render).
- **Fails closed.** Selecting `firebase` without working config/SDK throws a clear startup error and
  refuses to silently fall back to local. The adapter glue is dynamically imported (`firebaseBackend`
  4 kB, `firebaseServices` 0.27 kB), and the heavy SDK now lives in its own **lazy `firebase-vendor`
  chunk** (566 kB / 167 kB gzip) that loads only in firebase mode — out of the eager first load (§6.1).
- **Security posture.** `firestore.rules` enforce per-user ownership, require the stored `userId`/`id`
  to match the path, gate learning-data writes on `email_verified`, keep attempts append-only, keep
  content read-only, and apply a default-deny catch-all. Firebase Auth adds email verification
  (send/resend/reload) enforced both client-side and in rules. `firebase.json` ships baseline security
  headers (CSP, HSTS, nosniff, Referrer-Policy, X-Frame-Options, Permissions-Policy).
- **Defensive persistence.** Both backends validate/normalize records; the local backend recovers from
  corrupt JSON, scopes access to the active session user, caches parsed reads, and never persists
  passwords. Firebase mastery/attempts writes use Firestore transactions (concurrency-safe; attempts
  reject overwrite).
- **CI & supply-chain hygiene.** Every push to `main`/`local` and every PR runs lint + build + test; a
  separate workflow runs `npm audit` (prod high/critical fails the build) and CodeQL
  (`security-extended`), plus a weekly schedule. Dependabot groups weekly npm and Actions updates.

## 6. Current Issues (concrete, present today)

### 6.1 Firebase SDK code-splitting — FIXED (was a first-load bundle regression)

**Resolved in `af9c591`.** Previously `vite.config.ts` used
`manualChunks(id) => id.includes('node_modules') ? 'vendor' : undefined`, which swept **every
dependency into one `vendor` chunk** — React, the Firebase Auth/Firestore SDK, and KaTeX. Because
React and KaTeX are static dependencies of the entry, `vendor` was **modulepreloaded and loaded
eagerly**, so every visitor — including default local-mode users — downloaded the ~1,015 kB /
304 kB-gzip chunk that contained the unused Firebase SDK on first load (first-load JS ~346 kB gzip).

`manualChunks` now routes Firebase and its transitive deps (`firebase`, `@firebase/*`, `@grpc/*`,
`protobufjs`, `@protobufjs/*`, `idb`) to a dedicated **`firebase-vendor`** chunk, and KaTeX to its own
**`katex-vendor`** chunk; everything else third-party stays in `vendor`. Since Firebase is reached
**only** through the dynamic `import()` in `App.tsx` (firebase mode), `firebase-vendor` is **not
modulepreloaded** (verified absent from `dist/index.html`) — it loads only when firebase mode is
selected. Eager first-load JS for the default local mode drops from ~**346 kB → ~179 kB gzip**
(`vendor` 59.65 + `katex-vendor` 77.25 + app `index` 42.35, gzip), matching the README's
"Firebase is lazy-loaded … stays out of the default local bundle" claim and helping N1.

### 6.2 KaTeX adds a meaningful first-render payload

Beyond the JS, KaTeX brings a **28.8 kB (7.9 kB gzip) stylesheet** and a **large font set** (the
build emits **59** KaTeX font files — ~20 per format across `.woff2`/`.woff`/`.ttf`, ~1,073 kB total
on disk). Browsers fetch only the glyph formats they need, and only when math renders, but the first
math-heavy screen can incur a brief
font fetch / flash of unstyled math. This is an acceptable trade for correct notation, but it should
be measured against N1 once deployed (and `font-display` / preloading the primary `.woff2` faces is
worth considering).

### 6.3 Documentation drift across three files (none mention KaTeX)

- **`README.md`** still describes "the first three authored lessons, with shell lessons reserving
  the rest," lists "Next Priorities: Fill the Lesson 4-6 shells," and its smoke test references a
  "Lesson 4 shell." All six lessons are fully authored — this understates the work. It also does not
  mention the new KaTeX equation rendering.
- **`BACKEND_ADAPTERS.md`** lists `mastery` as a "per-skill EWMA score," but both backends store a
  plain cumulative ratio (§6.5).
- **`SECURITY.md`** says "No hosted backend yet. Firebase and Supabase adapters are not wired
  into the runtime," which contradicts the implemented, env-selectable `FirebaseBackend`.

### 6.4 Equation-rendering converter tests — FIXED (was untested)

**Resolved in `af9c591`.** `equationToLatex` and `equationToAriaLabel` (pure string→string transforms
with several regex branches) shipped in `9f8ed3c` with no tests. `tests/equation-latex.test.ts` now
locks in their behavior across 13 cases: pass-through of valid notation, `->` → `\rightarrow`,
`a/b` → `\frac`, prose → `\text`, `*` → `\cdot`, stray slashes left alone, empty/whitespace/nullish
input, and the `', then'` aria label — **plus security regression cases** proving command-like words
(`\href`, `\url`) are wrapped in `\text{}` and never survive as active commands. (`src/equationLatex.ts`
was added to `tsconfig.test.json`'s `include` so it compiles for the test run.) The DOM/`katex.render`
path inside `MathText` still needs a JSDOM/RTL harness (§ "No interaction-layer tests").

### 6.5 Mastery model: cumulative ratio, not EWMA; coarse skill attribution

Both `LocalBackend` and `FirebaseBackend` store `score = Math.round((correct / attempts) * 100) / 100`
(no recency weighting/decay), so early misses count forever and the model can't support the Phase-3
decay/spaced-repetition plan unchanged. Separately, `App.tsx` updates mastery for **every** `skillId`
of the active lesson with the same signal (`activeLesson.skillIds.map(... updateSkillMastery ...)`),
so multi-skill lessons (e.g. Balancing Equations → `equality` + `inverse-operations`) always show
identical per-skill scores.

### 6.6 MCQ logic lives in the view and duplicates engine escalation

Unlike the eight step types with engine checkers, there is no `checkMcqStep`; correctness
(`option.id === correctId`) and the hint→explanation→reveal escalation are reimplemented inline in
`MultipleChoiceStep`. A code comment even notes it "mirrors the engine's choice-step escalation," so
the two can drift apart.

### 6.7 Dead `CourseLessonNode.status` field (now also an unreachable UI state)

Each lesson node in `content/course.ts` carries `status: 'available' | 'locked'`, and the type in
`content/types.ts` was widened to include `'coming-soon'` — but the UI **never reads this field**.
Lock/unlock is computed dynamically from progress + prerequisites, and `comingSoon` is derived from
`lesson.steps.length === 0`, which is **always false** now that all six lessons are authored. So both
the `status` field and the "Coming soon" path-status branch (`getPathStatus`) are dead/unreachable
and can drift from reality.

### 6.8 No React error boundary

Async flows set a user-visible `runtimeError` banner (good), but a render-time exception still has no
error boundary to contain it (and the new `MathText` adds a render-time third-party path). Runtime
monitoring (Crashlytics / Performance, per the PRD) is also not wired.

### 6.9 `App.tsx` is a ~4162-line monolith (and growing)

It holds every screen and every step renderer plus the drag engine, and it grew ~170 lines in the
latest **code** commit (`9f8ed3c`). This is hard to navigate, review, and reuse; it is the natural
next refactor and also the code with the least automated coverage.

### 6.10 Toolchain Node-version sensitivity

The repo pins Node 24.16.0, and `scripts/node24-keepalive-fix.cjs` documents a **Node 24.17.0**
regression that breaks `firebase deploy` (it disables keep-alive as a workaround). Anyone deploying
must use the wrapper, and the pinned/CI Node should be kept off the broken patch. (This review ran
fine on Node 22.14, so local dev is tolerant, but CI and deploy are version-sensitive.)

## 8. Potential / Future Issues (risks to watch)

- **Cross-device sync/conflict semantics.** Firestore writes overwrite whole `LessonProgress` /
  `SkillMastery` documents, so two devices editing offline can clobber each other (last-write-wins).
  Decide merge vs. last-write semantics before heavy multi-device use (the BrainLift flags the same
  question about Brilliant itself).
- **CSP not validated against a live deploy.** The README explicitly TODOs verifying the CSP, replacing
  `style-src 'unsafe-inline'`, and HSTS `preload` after a real deploy; an untested CSP can silently
  break Auth/Firestore traffic or provide false confidence. (KaTeX injects inline styles, which
  interacts with any move away from `'unsafe-inline'`.)
- **COPPA / age gating unresolved** (PRD open question) for a teen audience — must be settled before any
  real account collection.
- **Bundle/content growth.** The Firebase SDK is now lazy and KaTeX is its own chunk (§6.1), so the
  eager first load is light again (~179 kB gzip), but KaTeX (§6.2) still adds ~77 kB gzip + fonts to it;
  content is bundled (good for now) but the PRD anticipates fetch-and-cache as the library grows.
- **Monitoring gap.** No Crashlytics/Sentry or Firebase Performance wiring yet, so production errors and
  N1/N2/N3 latency would be invisible after launch.
- **Mobile drag uses point hit-testing.** The keyboard/tap fallback mitigates accessibility, but the
  pointer-drag path (recently reworked) still warrants real iOS/Android Safari/Chrome testing.
- **No interaction-layer tests.** CI guards the pure logic, but a regression in the large `App.tsx`
  (drag/plot/slider/dragTerms) or in `MathText` rendering would pass CI undetected until manual QA.
