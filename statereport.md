# Balance — Project State Report

_Last generated: 2026-06-23 · Branch: `cursor/project-state-report-d82e` · Generated by automated codebase review_

This report summarizes the current state of the **Balance** project (a Brilliant-style,
learn-by-doing algebra MVP), what works, what is missing, current and potential issues, and
prioritized recommendations. It is based on a full read of the source, tests, configuration, and
the project docs (`Brilliant Practice PRD.md`, `Brilliant Backend BrainLift.md`, `README.md`,
`BACKEND_ADAPTERS.md`, `PHASE1_QA_CHECKLIST.md`).

---

## 1. Executive Summary

- **The local MVP is healthy and functional.** Build, lint, and the full test suite all pass
  (62/62 tests). The flagship balance-scale interaction, six fully authored lessons, wrong-answer
  recovery, local persistence, scoring, and a mastery-aware path are all implemented and tested.
- **The single biggest gap is the hosted backend.** Firebase is scaffolded but **not wired into
  the runtime**. As a result several PRD **P0** requirements are unmet: cross-device persistence,
  persistent password auth, a public HTTPS deployment, per-user security enforcement at runtime,
  and concurrent-user readiness. This is a known, documented deferral — but it is the gating work
  for the PRD's own Phase 1 hard gate.
- **Documentation has drifted from the code.** The README, QA checklist, and adapter notes still
  describe Lessons 4–6 as "shells" / "coming soon," but all six lessons are fully built and tested.
  These docs should be updated to avoid misleading reviewers.
- **A concrete accessibility bug exists:** the only drag-based balance step (Lesson 1's
  `drag-to-level`) has **no keyboard/tap fallback**, which contradicts the QA checklist's
  keyboard-fallback claim.

Overall: the learn-by-doing core is in good shape; the project is in the "wire the backend, deploy,
and reconcile docs" stage rather than a "core experience" stage.

## 2. Project Overview

**Product.** "Balance" teaches a single coherent path — _Algebra Foundations_ — through
hands-on interactive problems with instant, authored feedback. The flagship interaction is a
manipulable balance scale. Persona: a new-to-algebra 8th grader, mobile-first.

**Tech stack (verified from `package.json` and configs):**

| Area | Choice |
| --- | --- |
| Framework | React 19.2 |
| Build/dev | Vite 8, TypeScript ~6.0 (project references) |
| Lint | ESLint 10 flat config + typescript-eslint + react-hooks/react-refresh plugins |
| Tests | `node:test` over TypeScript compiled to `dist-tests/` |
| Backend (active) | `LocalBackend` — browser `localStorage` + `sessionStorage` |
| Backend (scaffolded) | Firebase 12 (Auth + Firestore + Hosting) — **not wired** |

**Architecture / key files:**

- `src/domain.ts` (~1.7k lines): all TypeScript types **and** the authored content (skills, the
  six lessons, the course). Content is typed TS objects (not external JSON).
- `src/engine.ts` (~560 lines): pure functions — answer checking per step type, scoring, mastery
  recommendation, progress transitions, path unlocking. This is the "client-side checker" the PRD
  requires.
- `src/backend.ts` (~490 lines): the app-owned `Backend` interface (`auth`, `progress`, `mastery`,
  `attempts`), `LocalBackend` implementation with input validation/sanitization, and a
  `createBackend(provider)` factory that **fails closed** for `firebase`.
- `src/App.tsx` (~1.8k lines): the entire UI — auth, course map, lesson player, all six step
  renderers, the balance-scale physics/drag handling, completion, and profile screens.
- `src/firebaseConfig.ts` / `src/firebaseConfigCore.ts`: env parsing and provider selection (used).
- `src/firebaseServices.ts`: Firebase SDK initialization — **currently unused by the app** (only
  referenced in docs); it is tree-shaken out of the build.
- `tests/engine.test.ts`, `tests/backend.test.ts`: 62 tests covering engine + backend logic.

**Architectural alignment with the BrainLift / PRD thesis:** the design correctly follows the
"backend is a sync/persistence layer; the smarts live client-side" model — answer checking,
scoring, and recommendations are all pure client-side functions. This is the right shape.

## 4. What Is Built (current state)

**Working and tested:**

- **Six fully authored lessons** (not shells): Balancing Equations, One-Step Equations, Two-Step
  Equations, Like Terms & Variables on Both Sides, Coordinate Plane, Graphing Lines. Each ends in a
  summary concept step; tests assert catalog/path coherence for all six.
- **Step types implemented:** `concept`, `mcq`, `input` (with numeric/equivalence checking),
  `operation-choice`, `sequence` (tap-to-order), and `balance` (the rich manipulative).
- **Flagship balance scale** with pointer-drag, apply-to-both-sides operations, real-time tilt, and
  level/isolate goal checking.
- **Authored, escalating feedback:** hint → explanation → reveal across attempts (`buildWrongResult`
  in `engine.ts`), with per-option/per-tile/per-answer misconception messages.
- **Progress + resume (local):** exact-step resume, per-lesson progress, completion history, latest
  & best first-try scores.
- **Mastery + path:** per-skill scores, prerequisite-based unlocking, "review suggested" vs
  "mastered" states, and a recommended next lesson.
- **Local auth:** email + display-name demo profiles (no passwords), session-scoped current user,
  cross-user access guards, and robust sanitization of corrupt/malformed `localStorage`.
- **Firebase fail-closed safety:** selecting `VITE_BACKEND_PROVIDER=firebase` before the adapter is
  ready surfaces a clear setup error instead of silently using local storage.
- **Accessibility groundwork:** ARIA roles/labels, `role="status"` feedback, a `prefers-reduced-motion`
  block in `App.css`, and non-color feedback text.

## 6. Current Issues

1. **Hosted backend not wired (highest impact).** `createBackend('firebase')` throws by design;
   `firebaseServices.ts` is unused. This blocks PRD P0 items N5, N9, N11, N12, N13 and weakens R1,
   R2, R25. Everything downstream (deploy, cross-device, concurrency) depends on this.

2. **Documentation drift — Lessons 4–6 described as shells.** All six lessons are fully authored and
   tested, but:
   - `README.md` ("shell lessons reserving the rest," "Lessons 4-6 kept as lightweight content
     shells," and the "Fill the Lesson 4-6 shells…" next priority),
   - `PHASE1_QA_CHECKLIST.md` (".. unlocks as a lightweight shell," "Lesson shells…"),
   - `BACKEND_ADAPTERS.md` (Phase 1 boundary mentions only Lesson 1/Lesson 2)
   
   all understate what is built. This is misleading for anyone evaluating the project.

3. **Accessibility: drag step has no keyboard/tap fallback.** In `App.tsx` `BalanceStepView`, the
   Left/Right "Place …" buttons render only when `!isPhysicalDrag`. The only step that actually has a
   draggable bank is Lesson 1's `drag-to-level`, which **is** `physical-drag` — so it exposes only a
   pointer-drag tile (`onPointerDown`) with no Enter/Space handler and no tap alternative. Keyboard-only
   users cannot complete it. This directly contradicts the QA checklist item "balance tile placement can
   be completed without pointer dragging by using the Left and Right buttons." (Touch works via pointer
   events; keyboard does not.) The non-physical fallback branch is effectively dead for current content.

4. **Mastery is per-lesson, not truly per-skill.** In `App.tsx` `completeStep`, every recorded attempt
   updates **all** of the lesson's `skillIds` with the same correct/incorrect value, regardless of which
   skill a given step exercises. Lessons tagged with two skills (Lesson 1: equality + inverse-operations;
   Lesson 4: like-terms + variables-on-both-sides) move both scores identically. Per-skill granularity is
   therefore coarser than the PRD implies.

5. **Mastery formula vs. docs mismatch (EWMA).** `BACKEND_ADAPTERS.md` and `README` say mastery is a
   "per-skill **EWMA** score," and the PRD floats EWMA as the MVP model. The actual implementation
   (`updateSkillMastery` in `backend.ts`) is a **cumulative lifetime accuracy** (`correct / attempts`),
   with no recency weighting. A learner who struggles early then improves keeps a depressed score, and
   retakes keep accumulating into the lifetime denominator — which can make "review suggested" sticky.

6. **Dead / unused code and types.**
   - `firebaseServices.ts` (`getFirebaseServices`) and, transitively, `getFirebaseConfig` are unused at
     runtime.
   - `CourseLessonNode.status` and the `'coming-soon'` literal in `domain.ts` are never read (lock/CTA
     state is computed dynamically in `App.tsx`).
   - The `!isPhysicalDrag` tap-fallback branch (see issue 3) never triggers with current content.

## 7. Potential Issues / Risks

- **PRD Phase 1 hard gate at risk on hosted items.** The PRD's Phase 1 checklist (its own gate)
  includes "Deployed and public," "progress persists across devices," and "holds up under multiple
  concurrent learners" as P0. These cannot be demonstrated without the backend + a deploy.
- **No code-splitting / lazy-loading (N1/N4).** The whole app is one ~280 kB chunk (84 kB gzip).
  Fine for the current size, but it diverges from the PRD's "course map and first lesson load first;
  heavier step components load on demand," and will matter once plot/slider/Firebase code lands.
- **No CI.** There is no `.github/workflows`. Tests, lint, and build exist but are not enforced on
  pushes/PRs, so regressions (e.g., the doc/code drift) can slip in.
- **No UI/component tests.** Coverage is strong for `engine.ts`/`backend.ts` but `App.tsx` (the
  largest, most stateful file — view state machine, pointer-drag, `elementFromPoint` hit-testing) is
  untested. Drag/keyboard regressions would not be caught.
- **Backend interface is synchronous.** `Backend` methods are sync because `LocalBackend` reads
  storage directly. Firebase is async; as `BACKEND_ADAPTERS.md` notes, swapping in `FirebaseBackend`
  requires converting the contract to async and threading loading/error states through React — a
  non-trivial, app-wide change, not a drop-in.
- **Performance targets unmeasured.** N3 (60 FPS) and N2 (<100ms) are architecturally sound but not
  instrumented (no Firebase Performance Monitoring, as the PRD envisions).
- **`App.tsx` and `domain.ts` size.** Both are very large single files. As content and interaction
  types grow, this will slow navigation and increase merge-conflict risk.
- **COPPA / age gating unresolved (N12, PRD open question).** The target audience is ~13; sign-up has
  no age handling. Relevant once real auth/PII is collected.

## 8. What Needs To Be Done (prioritized)

**Priority 1 — Reconcile docs with reality (low effort, high clarity).**
- Update `README.md`, `PHASE1_QA_CHECKLIST.md`, and `BACKEND_ADAPTERS.md` to state that all six
  lessons are built and tested; remove "shell"/"coming soon" language and the stale "fill the
  Lesson 4-6 shells" priority.
- Fix the mastery description (cumulative average, not EWMA) **or** implement EWMA to match the docs.

**Priority 2 — Close the accessibility gap (small, well-scoped).**
- Give the `physical-drag` balance step a keyboard/tap fallback (e.g., reuse the Left/Right "Place"
  buttons, or add Enter/Space activation), so Lesson 1 is completable without a pointer. Then the QA
  checklist's keyboard-fallback claim becomes true.

**Priority 3 — Wire the Firebase backend (the main feature work; unblocks PRD P0).**
- Introduce an async provider factory; convert the `Backend` contract to async and handle
  loading/error states in React.
- Implement `FirebaseBackend` (Auth email/password sign-up/in/out + display name; Firestore
  `users/`, `progress/`, `mastery/`, `attempts/` matching `firestore.rules`).
- Add a one-time local→Firestore migration path for existing demo progress (optional).
- Validate `firestore.rules` against the live data model (rules are already authored and look
  correct: per-user read/write, content read-only, default deny).

**Priority 4 — Deploy and verify hosted P0s.**
- `firebase use --add`, enable Email/Password auth + Firestore + Hosting, then
  `npm run build && firebase deploy`. This satisfies N13 and enables N5/N9/N11/N12 verification.
- Run the deferred hosted checks in `PHASE1_QA_CHECKLIST.md` (cross-device resume, rules enforcement,
  concurrent-user smoke test).

**Priority 5 — Engineering hardening.**
- Add CI (lint + test + build on PR).
- Add component/interaction tests for `App.tsx` (drag, keyboard, view transitions).
- Add code-splitting/lazy-loading for heavier step components and the Firebase adapter.

## 9. Ways To Improve

- **Make mastery per-skill-per-step.** Tag individual steps with the skill(s) they exercise and
  update only those, rather than all lesson skills uniformly. Consider EWMA so recent performance and
  retakes are reflected fairly.
- **Build the remaining rich manipulatives (PRD P1):** an interactive coordinate grid (plot points,
  R15), m/b sliders driving a live line (R16), and drag-term tiles across the `=` sign with automatic
  sign change (R14). These would raise Lessons 4–6 from "tap/order proxies" to true direct
  manipulation, matching the product thesis.
- **Split large files.** Extract step renderers and the balance-scale component(s) out of `App.tsx`;
  separate content data from type definitions in `domain.ts` (or move content to bundled JSON per the
  PRD's data-driven vision).
- **Remove or activate dead code:** either wire `firebaseServices.ts` (Priority 3) or drop it until
  needed; remove the unused `CourseLessonNode.status` / `'coming-soon'` type or actually drive UI
  from it.
- **Instrument performance** (first-interaction, feedback latency, FPS) to back the N1/N2/N3 claims —
  Firebase Performance Monitoring once the backend is live.
- **Add analytics for content quality** (step-level wrong-answer rates) to find weak problems, as the
  PRD's measurement section describes.
- **Resolve COPPA/age-gating** before collecting real credentials/PII.
