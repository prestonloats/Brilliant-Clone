coherent path — **Algebra Foundations** — through interactive, typed lesson steps with instant,
authored feedback. The flagship interaction is a manipulable **balance scale**, now joined by an
interactive coordinate grid, slope/intercept sliders, term-tile sorting, and creative

## 2. What the Project Is (Context)

- **Subject:** Algebra only — a deliberate depth-over-breadth bet (PRD "Non-Goals").
- **Persona:** "Maya," a new-to-algebra 8th grader, mobile-first.
- **Thesis:** Every lesson is a short sequence of *typed, interactive steps* rendered from a data
  model, with sub-100 ms client-side answer checking and specific authored feedback for both right
  and common wrong answers — never a bare red ✗.
- **Course path — now a branching DAG** (not a straight line): Balancing Equations → One-Step →
  Two-Step, then Two-Step **diverges** into both **Like Terms** and **Coordinate Plane**, which
  **merge** as prerequisites of **Graphing Lines**. `prerequisites` on each lesson encode the graph;
  `lessonOrder` keeps a valid topological order for recommendations.

## 3. Architecture & Tech Stack

**Stack:** React 19.2 + TypeScript ~6.0 + Vite 8, ESLint 10 / typescript-eslint 8, Node's built-in
test runner. `firebase` v12.15 is now a real runtime dependency (lazy-loaded). No CSS framework
(hand-written `App.css`, ~49 kB).

**Module layout (`src/`):**

| File / dir | Lines | Responsibility |
| --- | --- | --- |
| `App.tsx` | ~3955 | Entire React UI: all screens, step renderers, drag/keyboard interactions, async data flow |
| `engine.ts` | ~982 | Pure answer-checkers for every step type + progress/score/mastery/branch-aware recommendation |
| `backend.ts` | ~532 | App-owned `Backend` contract (now async-capable) + `LocalBackend` + provider factory + validators |
| `firebaseBackend.ts` / `firebaseBackendCore.ts` | ~281 / ~75 | `FirebaseBackend` (Auth + Firestore) + path/owner/verify helpers |
| `content/` | per-file | `types.ts` (content model), `course.ts`, `skills.ts`, `lessons/*.ts` (one file per lesson), `examples/` (authoring sample) |
| `domain.ts` | ~80 | Barrel re-exporting content types/data + runtime types (profile, progress, mastery, attempts) |
| `authValidation.ts` | ~58 | Pure, provider-neutral email/password form validation (shared by UI + tests) |
| `firebaseConfig*.ts` / `firebaseServices.ts` | ~120 | Env-driven Firebase config + SDK init (lazy) |

**Content model (`content/types.ts`):** A `Lesson` is an ordered array of tagged-union `LessonStep`s.
Step types now number **ten**: `concept`, `mcq`, `input`, `balance`, `operation-choice`, `sequence`,
plus the four new hands-on types **`manipulative`** (equal-groups / collect puzzles), **`plot`**
(interactive coordinate grid), **`slider`** (slope/intercept → live line), and **`dragTerms`**
(term-tile sorting). Per-lesson step usage:

- **Balancing Equations:** 3 concept, 1 mcq, 2 input, **3 balance**, 2 sequence.
- **One-Step Equations:** 3 concept, 4 input, 1 balance, 1 operation-choice, 2 sequence, **1 manipulative**.
- **Two-Step Equations:** 2 concept, 3 input, 1 balance, 2 operation-choice, 3 sequence, **1 manipulative**.
- **Like Terms & Variables Both Sides:** 2 concept, 2 input, 1 operation-choice, 3 sequence, **3 dragTerms**.
- **Coordinate Plane:** 3 concept, 2 input, 1 sequence, **4 plot**.
- **Graphing Lines:** 2 concept, 2 input, 2 operation-choice, 1 sequence, **2 slider**.

**Design strengths:**

- **Clean separation of concerns.** Content (`content/`), pure logic (`engine.ts`), and persistence
  (`backend.ts` / `firebaseBackend.ts`) are decoupled; the engine functions are pure and well tested.
  Lessons are now one file each, which supports parallel authoring.
- **Provider-neutral backend behind one contract.** React talks only to `Backend`; both `LocalBackend`
  and `FirebaseBackend` implement it. The contract is **async-capable** (`MaybePromise`), and the UI
  loads learner data into React state and handles loading/error states (no more reads during render).
- **Fails closed.** Selecting `firebase` without working config/SDK throws a clear startup error and
  refuses to silently fall back to local. Firebase is **lazy-loaded** via dynamic `import()`, so the
  ~566 kB SDK stays out of the default local bundle.
- **Security posture.** `firestore.rules` enforce per-user ownership, require the stored `userId`/`id`
  to match the path, gate learning-data writes on `email_verified`, keep attempts append-only, and
  keep content read-only with a default-deny catch-all. Firebase Auth adds email verification
  (send/resend/reload) enforced both client-side and in rules. `firebase.json` ships baseline security
  headers (CSP, HSTS, nosniff, Referrer-Policy, X-Frame-Options, Permissions-Policy).
- **Defensive persistence.** Both backends validate/normalize records; the local backend recovers from
  corrupt JSON, scopes access to the active session user, and never persists passwords. Firebase
  mastery/attempts writes use Firestore transactions (concurrency-safe; attempts reject overwrite).

## 6. Current Issues (concrete, present today)

1. **Documentation drift (README + BACKEND_ADAPTERS).** `README.md` still describes the project as
   completing "the first three authored lessons, with shell lessons reserving the rest" (intro, the
   MVP smoke-test steps, and "Next Priorities: Fill the Lesson 4-6 shells"), even though all six
   lessons are fully authored with rich interactions. `BACKEND_ADAPTERS.md` still lists `mastery` as a
   "per-skill EWMA score," but the implementation is a plain cumulative ratio. These understate the
   work done and misdescribe the mastery model.

2. **Mastery model: cumulative ratio, not EWMA; coarse skill attribution.** Both `LocalBackend` and
   `FirebaseBackend` store `score = correct / attempts` (no recency weighting/decay), so early misses
   count forever and the model can't support the Phase-3 decay/spaced-repetition plan unchanged.
   Separately, `App.tsx` updates mastery for **every** `skillId` of the active lesson with the same
   signal (`activeLesson.skillIds.map(... updateSkillMastery ...)`), so multi-skill lessons (e.g.
   Balancing Equations → `equality` + `inverse-operations`) always show identical per-skill scores.

3. **MCQ logic lives in the view and is untested by the engine.** Unlike the other nine step types,
   there is no `checkMcqStep`; MCQ correctness/escalation is implemented inline in `MultipleChoiceStep`
   in `App.tsx` and is not covered by engine unit tests.

4. **Dead `CourseLessonNode.status` field.** Each lesson node in `content/course.ts` carries
   `status: 'available' | 'locked'`, but the UI computes lock/unlock dynamically from progress +
   prerequisites and never reads this field. It is dead data that can drift from reality.

5. **Large Firebase vendor chunk.** The lazy Firebase chunk is ~566 kB (167 kB gzip) and trips Vite's
   ">500 kB" warning. It is correctly code-split (no impact on local mode), but firebase-mode users
   download a heavy bundle; it could be split further (auth vs. firestore) or deferred more granularly.

6. **No CI and no React-layer/interaction tests.** There is no `.github/` workflow, so lint/test/build
   aren't enforced on push, and the React UI (including the balance/plot/slider/dragTerms interactions)
   has no automated coverage beyond the pure checkers and the manual QA checklist.

7. **No React error boundary.** Async flows now set a user-visible `runtimeError`, which is good, but a
   render-time exception still has no error boundary to contain it; runtime monitoring (Crashlytics /
   Performance, per the PRD) is also not wired.

8. **`App.tsx` is a ~3955-line monolith.** It holds every screen and every step renderer plus the drag
   engine. This is becoming hard to navigate, review, and reuse; it is the natural next refactor.

## 8. Potential / Future Issues (risks to watch)

- **Cross-device sync/conflict semantics.** Firestore writes overwrite whole `LessonProgress` /
  `SkillMastery` documents, so two devices editing offline can clobber each other (last-write-wins).
  Decide merge vs. last-write semantics before heavy multi-device use (the BrainLift flags the same
  question about Brilliant itself).
- **CSP not validated against a live deploy.** The README explicitly TODOs verifying the CSP, replacing
  `style-src 'unsafe-inline'`, and HSTS `preload` after a real deploy; an untested CSP can silently
  break Auth/Firestore traffic or provide false confidence.
- **COPPA / age gating unresolved** (PRD open question) for a teen audience — must be settled before any
  real account collection.
- **Bundle/content growth.** Firebase vendor weight plus a single large app chunk means first-load will
  grow; content is bundled (good for now) but the PRD anticipates fetch-and-cache as the library grows.
- **Monitoring gap.** No Crashlytics/Sentry or Firebase Performance wiring yet, so production errors and
  N1/N2/N3 latency would be invisible after launch.
- **Mobile drag uses point hit-testing.** The keyboard/tap fallback mitigates accessibility, but the
  pointer-drag path still warrants real iOS/Android Safari/Chrome testing.
