## 2. What the Project Is (Context)

- **Subject:** Algebra only — a deliberate depth-over-breadth bet (PRD "Non-Goals").
- **Persona:** "Maya," a new-to-algebra 8th grader, mobile-first.
- **Thesis:** Every lesson is a short sequence of *typed, interactive steps* rendered from a data
  model, with sub-100 ms client-side answer checking and specific authored feedback for both right
  and common wrong answers — never a bare red ✗.
- **Course path — a branching DAG** (not a straight line): Balancing Equations → One-Step →
  Two-Step, then Two-Step **diverges** into both **Like Terms** and **Coordinate Plane**, which
  **merge** as prerequisites of **Graphing Lines**. `prerequisites` on each lesson encode the graph;
  `lessonOrder` keeps a valid topological order for recommendations.

## 3. Architecture & Tech Stack

**Stack:** React 19.2 + TypeScript ~6.0 + Vite 8 (Rolldown), ESLint 10 / typescript-eslint 8,
Node's built-in test runner. `firebase` v12.15 and **`katex` v0.17** (+ `@types/katex`) are real
runtime dependencies. Firebase glue is dynamically imported by the adapter, but both the Firebase

`App.css` plus KaTeX's stylesheet). Dev tooling pins **Node 24.16.0** (`.nvmrc`, `.node-version`,
CI); this review ran on Node 22.14 and all checks still passed.

**Content model (`content/types.ts`):** A `Lesson` is an ordered array of tagged-union `LessonStep`s.
Step types number **ten**: `concept`, `mcq`, `input`, `balance`, `operation-choice`, `sequence`,
plus four hands-on types — **`manipulative`** (equal-groups / collect / **new** build-product
"discover-the-total" puzzles), **`plot`** (interactive coordinate grid), **`slider`**
(slope/intercept → live line), and **`dragTerms`** (term-tile sorting). Per-lesson step usage (each
lesson file is fully authored — none are "shells"):

- **Balancing Equations** (277 lines), **One-Step** (282), **Two-Step** (361), **Like Terms &
  Variables on Both Sides** (321), **The Coordinate Plane** (192), **Graphing Lines** (218).

**Design strengths:**

- **Clean separation of concerns.** Content (`content/`), pure logic (`engine.ts`), and persistence
  (`backend.ts` / `firebaseBackend.ts`) are decoupled; the engine functions are pure and well tested.
  Lessons are one file each, which supports parallel authoring.
- **Provider-neutral backend behind one contract.** React talks only to `Backend`; both `LocalBackend`
  and `FirebaseBackend` implement it. The contract is **async-capable** (`MaybePromise`), and the UI
  loads learner data into React state and handles loading/error states (no reads during render).
- **Fails closed.** Selecting `firebase` without working config/SDK throws a clear startup error and
  refuses to silently fall back to local. The adapter glue is dynamically imported (`firebaseBackend`
  4 kB, `firebaseServices` 0.26 kB), though the SDK itself currently rides the eager vendor chunk.
- **Accessible math rendering.** `MathText` renders KaTeX into a ref with `throwOnError:false` and a
  plain-text fallback; the visible output is `aria-hidden` while a visually-hidden span carries the
  original equation, so screen readers (and aria-live regions) announce real meaning, not markup.
- **Security posture.** `firestore.rules` enforce per-user ownership, require the stored `userId`/`id`
  to match the path, gate learning-data writes on `email_verified`, keep attempts append-only, keep
  content read-only, and apply a default-deny catch-all. Firebase Auth adds email verification
  (send/resend/reload) enforced both client-side and in rules. `firebase.json` ships baseline security
  headers (CSP, HSTS, nosniff, Referrer-Policy, X-Frame-Options, Permissions-Policy).
- **Defensive persistence.** Both backends validate/normalize records; the local backend recovers from
  corrupt JSON, scopes access to the active session user, caches parsed reads, and never persists
  passwords. Firebase mastery/attempts writes use Firestore transactions (concurrency-safe; attempts
  reject overwrite).
- **CI & supply-chain hygiene.** Every push to `main`/`local` and every PR runs lint + build + test;
  a separate workflow runs `npm audit` (prod high/critical fails the build) and CodeQL
  (`security-extended`), plus a weekly schedule. Dependabot groups weekly npm and Actions updates.

## 6. Current Issues (concrete, present today)

### 6.1 First-load regression worsened — Firebase SDK **and now KaTeX** ride the eager vendor chunk

`vite.config.ts` uses `manualChunks(id) => id.includes('node_modules') ? 'vendor' : undefined`. The
intent (cache the React runtime separately) is reasonable, but it sweeps **every** third-party module
into one `vendor` chunk. Because React is a static dependency of the entry, `vendor` is
**modulepreloaded and loaded eagerly** (confirmed in `dist/index.html`: `<link rel="modulepreload"
… vendor-*.js>` and the matching `vendor-*.css`). Net effect today:

- The unused **Firebase Auth/Firestore SDK** is downloaded by every visitor, including default
  local-mode users — the dynamic `import()` of the adapter now only splits ~4 kB of glue, not the
  heavy SDK it was meant to defer.
- **KaTeX is now in the same boat.** `src/main.tsx` statically imports `katex/dist/katex.min.css`
  and `src/MathText.tsx` statically imports `katex`, so KaTeX's JS lands in `vendor` and its CSS in
  the eager `vendor-*.css`.

Combined, the vendor chunk grew from **757 kB → 1,015.5 kB** (227 → **304 kB gzip**), pushing
first-load JS from ~268 kB gzip to **~346 kB gzip**, plus a new ~7.9 kB-gzip KaTeX CSS file. This
contradicts the README ("Firebase is lazy-loaded … stays out of the default local bundle") and works
against N1. **Fix:** give Firebase (and optionally KaTeX) their own manual chunks, or drop the blanket
`node_modules → vendor` rule, so the SDK only loads in firebase mode and KaTeX can be split/deferred.

### 6.2 Documentation drift across three files

- **`README.md`** still describes "the first three authored lessons, with shell lessons reserving
  the rest," lists "Next Priorities: Fill the Lesson 4-6 shells," and its smoke test references a
  "Lesson 4 shell." All six lessons are fully authored — this understates the work. It also predates
  the KaTeX equation rendering.
- **`BACKEND_ADAPTERS.md`** lists `mastery` as a "per-skill EWMA score," but both backends store a
  plain cumulative ratio (§6.3).
- **`SECURITY.md`** says "No hosted backend yet. Firebase and Supabase adapters are not wired into
  the runtime," which contradicts the implemented, env-selectable `FirebaseBackend`.

### 6.3 Mastery model: cumulative ratio, not EWMA; coarse skill attribution

Both `LocalBackend` (`backend.ts:441`) and `FirebaseBackend` (`firebaseBackend.ts:203`) store
`score = round(correct / attempts * 100) / 100` (no recency weighting/decay), so early misses count
forever and the model can't support the Phase-3 decay/spaced-repetition plan unchanged. Separately,
`App.tsx` (`activeLesson.skillIds.map(... updateSkillMastery ...)`, ~line 271) updates mastery for
**every** `skillId` of the active lesson with the same signal, so multi-skill lessons (e.g. Balancing
Equations → `equality` + `inverse-operations`) always show identical per-skill scores.

### 6.4 MCQ logic lives in the view and duplicates engine escalation

Unlike the other nine step types, there is no `checkMcqStep`; correctness (`option.id === correctId`)
and the hint→explanation→reveal escalation are reimplemented inline in `MultipleChoiceStep` in
`App.tsx`. A code comment notes it "mirrors the engine's choice-step escalation," so the two can
drift apart.

### 6.5 Dead `CourseLessonNode.status` field

Each lesson node in `content/course.ts` carries `status: 'available' | 'locked'` (the type now also
allows `'coming-soon'`), but the UI computes lock/unlock dynamically from progress + prerequisites and
the "Coming soon" label is derived from `lesson.steps.length === 0` (`App.tsx:1034`), **not** from
this field. It remains dead data that can drift from reality.

### 6.6 No React error boundary

Async flows set a user-visible `runtimeError` banner (good), but a render-time exception still has no
error boundary to contain it (`main.tsx` renders `<App />` directly under `StrictMode`); runtime
monitoring (Crashlytics / Performance, per the PRD) is also not wired.

### 6.7 `App.tsx` is a ~4162-line monolith

It holds every screen and every step renderer plus the drag engine, and it keeps growing (it gained
~170 lines in the latest commit). This is hard to navigate, review, and reuse; it is the natural next
refactor and also the code with the least automated coverage.

### 6.8 Toolchain Node-version sensitivity

The repo pins Node 24.16.0, and `scripts/node24-keepalive-fix.cjs` documents a **Node 24.17.0**
regression that breaks `firebase deploy` (it disables keep-alive as a workaround). Anyone deploying
must use the wrapper, and the pinned/CI Node should be kept off the broken patch. (This review ran
fine on Node 22.14, so local dev is tolerant, but CI and deploy are version-sensitive.)

## 8. Potential / Future Issues (risks to watch)

- **CSP vs. KaTeX inline styles.** KaTeX's HTML output relies on inline `style` attributes, so it
  currently depends on the CSP's `style-src 'unsafe-inline'`. The README's TODO to *remove*
  `'unsafe-inline'` (switch to hashes/nonces) would now **break math rendering** unless KaTeX is
  accounted for — factor this in before tightening the CSP.
- **Cross-device sync/conflict semantics.** Firestore writes overwrite whole `LessonProgress` /
  `SkillMastery` documents, so two devices editing offline can clobber each other (last-write-wins).
  Decide merge vs. last-write semantics before heavy multi-device use (the BrainLift flags the same
  question about Brilliant itself).
- **CSP not validated against a live deploy.** The README explicitly TODOs verifying the CSP, replacing
  `style-src 'unsafe-inline'`, and HSTS `preload` after a real deploy; an untested CSP can silently
  break Auth/Firestore traffic or provide false confidence.
- **COPPA / age gating unresolved** (PRD open question) for a teen audience — must be settled before any
  real account collection.
- **Bundle/content growth.** With the SDK and KaTeX now eager (§6.1), first-load is already heavy;
  content is bundled (good for now) but the PRD anticipates fetch-and-cache as the library grows.
- **Monitoring gap.** No Crashlytics/Sentry or Firebase Performance wiring yet, so production errors and
  N1/N2/N3 latency would be invisible after launch.
- **Mobile drag uses point hit-testing.** The keyboard/tap fallback mitigates accessibility, but the
  pointer-drag path still warrants real iOS/Android Safari/Chrome testing.
- **No interaction-layer tests.** CI guards the pure logic, but a regression in the large `App.tsx`
  (drag/plot/slider/dragTerms/KaTeX) would pass CI undetected until manual QA.
