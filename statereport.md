## 1. Executive Summary

**Balance** is a "learn-by-doing" algebra web app modeled on Brilliant.org. Phase 1 teaches a single
coherent path — **Algebra Foundations** (six fully authored lessons) — through interactive, typed
lesson steps with instant, authored, client-side feedback (flagship interaction: a manipulable
**balance scale**, joined by a coordinate grid, slope/intercept sliders, term-tile sorting, and
"manipulative" puzzles; equations typeset with **KaTeX**). On top of that core now sit **Story Mode**
(Phase 2, the first AI feature) and a **learning-science practice engine** (Phase 3).

## 2. What the Project Is (Context)

- **Subject:** Algebra only — a deliberate depth-over-breadth bet (PRD "Non-Goals").
- **Persona:** "Maya," a new-to-algebra 8th grader, mobile-first.
- **Thesis (Phase 1):** Every lesson is a short sequence of *typed, interactive steps* rendered from a
  data model, with sub-100 ms client-side answer checking and specific authored feedback for both right
  and common wrong answers — never a bare red ✗.
- **Course path — a branching DAG:** Balancing Equations → One-Step → Two-Step, then Two-Step
  **diverges** into both **Like Terms** and **Coordinate Plane**, which **merge** as prerequisites of
  **Graphing Lines**. `prerequisites` encode the graph; `lessonOrder` keeps a topological order.
- **Story Mode (Phase 2):** an endless, interest-themed choose-your-own-adventure that **re-themes**
  code-graded practice questions and narrates between them — **pure review** of already-unlocked skills,
  built on the same content model and client-side checker. Unlocks after the first two lessons.
- **Learning science (Phase 3):** Story Mode is the **practice surface** where retrieval practice,
  spaced repetition, interleaving, and mastery learning make the material stick.

The PRD (dated 2026-06-22) anticipated all three phases; this report finds Phase 1 complete, Phase 2
shipped, and Phase 3 implemented for real (not cosmetic).

**Story-Mode design strengths (the important architectural facts):**

- **Grading is code-authoritative.** Each of the 9 architectures purely `generate(rng) → { step, answer }`
  from a `paramSeed`; the LLM `rethemeQuestion` only rewrites display labels; `applyRetheme` overwrites
  *labels only* (never `accept`/`correctId`); `themedCoherence` rejects a rewrite whose numbers drift;
  on failure the canonical un-themed question is shown. Resume rebuilds questions from
  `architectureId + paramSeed`, never trusting stored answers. The sub-100 ms client-side checker is
  reused unchanged.
- **Fails soft at every AI call.** Theme-aware offline fallbacks for beats/rethemes; new adventures need
  a configured provider, but saved adventures **resume offline**.
- **Pure/impure split.** Reducers, selectors, scheduler, and validators are React-free and heavily
  tested; `useStorySession` is the only React glue.
- **Pure-review invariant (narrowed).** Story Mode **never** writes `LessonProgress` or the lesson
  `mastery` ratio; it writes only the **dedicated `practice` store** and `source:'story'` attempt
  events, so course unlocks/grades are unaffected.

**Learning-science (Phase 3) facts:** one `SkillPracticeState` per user-per-skill (proficiency EWMA,
streak, SM-2 `ease`/`intervalDays`/`dueAt`/`lapses`, counters). `selectNextArchitecture` composes:
lesson gate → mastery-prereq gate → anti-repeat window → **hard interleaving** → **due-first** → weighted
pick (struggle ×2, mastered ×0.75, overdue boost ≤3, recent-miss ×1.5). Mastery =
`proficiency ≥ 0.9 AND streak ≥ 5`. Insights expose per-skill mastery bars and a "did it stick?"
retention metric (first-exposure vs. spaced re-exposure accuracy/latency).

## 6. Current Issues (concrete, present today)

### 6.1 Heavy SDKs (Firebase + OpenAI + Gemini) ride the eager first-load chunk — regression worsened

`vite.config.ts` still uses `manualChunks(id) => id.includes('node_modules') ? 'vendor' : undefined`,
which sweeps **every** dependency into one `vendor` chunk: React, the Firebase SDK, KaTeX, **and now the
`openai` and `@google/genai` SDKs**. Because React/KaTeX are static entry deps, `vendor` is
`modulepreload`ed and loaded eagerly (verified in `dist/index.html`). Net effect: **every visitor,
including the default local-mode demo user who never opens Story Mode, downloads a 1,531.95 kB /
411.58 kB-gzip chunk** containing the unused Firebase + AI SDKs. The dynamic `import()` seams only split
tiny adapter *glue* (0.5–3 kB), not the heavy SDKs. First-load JS is now ~**517 kB gzip** (vendor 412 +
app 106), working hard against N1. **Fix:** give Firebase and the AI SDKs their own manual chunks (so
they load only when actually used), and optionally split KaTeX for caching.

### 6.2 The Story-Mode AI proxy (Cloudflare Worker) is effectively an open, unauthenticated proxy

`worker/worker.js` gates requests only by an **Origin allow-list** (which it correctly notes is *not* a
trust boundary — it is trivially spoofable by non-browser clients) plus an **optional** rate limiter
(`if (env.STORY_RATE_LIMITER)` — 60 req/min/IP when bound) and input validation (op + model allow-list +
size caps). There is **no caller authentication** — no Firebase App Check, no auth/ID-token check. Anyone
who learns the worker URL (it is baked into the client build / referenced in the CSP) can call OpenAI on
the project's key within those bounds, i.e. a **wallet-drain / abuse** risk. The **Firebase AI Logic**
path is stronger (App Check is the intended gate) but App Check enablement is an unfinished operator
task. **Fix:** require an App Check / Firebase ID token at the worker, keep the rate limiter mandatory,
and set a hard OpenAI spend cap as a backstop. (Note: `.env.example`/comments also reference a Cloud
Function `functions/` proxy that does not exist in the repo — only the worker and dev plugin do.)

### 6.3 AI content safety for a minor audience is layered but best-effort

Safety is genuinely multi-layered (system-prompt constraints → provider safety settings, e.g. Gemini
`BLOCK_LOW_AND_ABOVE` → input sanitize/profanity/unsafe filters → OpenAI Moderations on raw choices →
output word-boundary filter → safe fallbacks), and `tests/story-safety.test.ts` (~60 kB) is one of the
stronger test files. **But** the local word lists are small and bypassable; an LLM can still emit subtle
inappropriate content that passes filters; **OpenAI moderation fails *open* on a client network error**
(`isModerationFlagged` returns `false` on catch); typed *interests*/cast text get UI-level validation but
less per-call runtime moderation than checkpoint choices; and there is no human review or age
verification. For a stated 8th-grade audience using LLM-generated narrative, this is an inherent residual
risk to track, not a solved problem.

  mastery is a cumulative ratio — EWMA exists only in the *practice* store), says content is "bundled in
  `src/domain.ts`," and omits the new `PracticeRepository` and story persistence.
- **Minor:** `LEARNING_SCIENCE.md` §9 cites "794 tests"; the suite is now **876**.

### 6.5 Two parallel mastery models; lesson mastery is still coarse

The app now has **two** mastery systems. The **lesson** `SkillMastery` is still
`Math.round((correct/attempts)*100)/100` — a cumulative ratio with no recency weighting — and the app
layer updates **every** `skillId` of the active lesson with the same signal (`LearningApp.tsx` ~214), so
multi-skill lessons always show identical per-skill scores. The **new** practice store
(`engine/practice/`) is a proper EWMA proficiency + SM-2 model with **per-skill** attribution (via the
question architecture's `skillId`). The two are intentionally separate (so practice can't corrupt lesson
grading), but it means the PRD's R26 mastery (lesson side) is still the naive model the docs once
promised to fix, while the sophisticated model lives only behind Story Mode.

### 6.6 MCQ checking + checkpoint-interval divergence from the engine/docs

Two "single source of truth" gaps: (1) there is still **no `checkMcqStep`** — correctness and the
hint→explanation→reveal escalation are reimplemented inline in `lesson/steps/MultipleChoiceStep.tsx`
(a comment notes it "mirrors the engine's choice-step escalation"), so the two can drift. (2) PRD **R35**
and `STORY_MODE_IMPLEMENTATION_PLAN.md` both say a checkpoint **every 10 solved questions**, but the code
uses `CHECKPOINT_INTERVAL = 5` (`storySessionReducer.ts`). The code may be the intended tuning, but the
docs and code should be reconciled.

### 6.7 Equation-rendering code still ships without tests

`equationToLatex` (a pure string→string transform with several regex branches) and `MathText` have **no
tests** — precisely the kind of pure function the repo unit-tests everywhere else. Edge cases (`x/6` vs.
a stray `/`, multi-letter variables, arrows, negatives, empty input) are worth pinning before lessons or
Story rethemes author more notation against it.

### 6.8 No React error boundary; no runtime monitoring

Async flows set a user-visible `runtimeError` banner, but there is still **no error boundary** to contain
a render-time exception — now a larger surface given KaTeX, the Story-Mode screens, and AI-driven content
rendering. Crashlytics / Firebase Performance (per the PRD) are not wired, so production errors and
N1/N2/N3 latency would be invisible after launch.

### 6.9 React/interaction layer and the worker remain untested

The pure logic is exhaustively tested, but: the lesson step renderers, the **large Story-Mode `.tsx`
screens**, and `MathText` have **no render tests** (no JSDOM/RTL); and the **Cloudflare `worker.js`
handler** (CORS, rate limit, validation, upstream error handling) has **no automated tests and is not in
CI** — only the shared `openAiProxyProtocol` is unit-tested. A regression in the proxy or the UI would
pass CI undetected until manual QA.

### 6.10 Password reset (R5) not implemented

Neither backend exposes a reset path. Firebase makes this near-free (`sendPasswordResetEmail`); without
it, a learner who forgets a password has no recovery in either mode. (P2 in the PRD.)

### 6.11 Persistent toolchain / supply-chain items

- **`firebase-tools` dev advisories:** 5 moderate, dev-only (`gaxios`→`uuid`, `@opentelemetry/core`,
  `@google-cloud/pubsub`). `npm audit --omit=dev` is clean and the CI prod gate (`--audit-level=high`)
  passes; `npm audit fix --force` would downgrade `firebase-tools` to 14.x (breaking) — track upstream
  instead, and scope `SECURITY.md`'s "zero" claim to production deps.
- **Node 24.17.0 deploy regression:** `scripts/node24-keepalive-fix.cjs` documents a Node 24.17.0 bug
  that breaks `firebase deploy`; keep the pinned/CI Node off that patch and use the wrapper.

## 8. Potential / Future Issues (risks to watch)

- **AI cost & abuse exposure.** With an unauthenticated worker proxy (§6.2) and free-tier reliance, a
  leaked/guessed worker URL or App Check gap is a direct billing/abuse risk. Spend caps + auth are the
  backstops.
- **Cross-device conflict semantics.** Firestore writes overwrite whole documents (now including whole
  `StorySession` docs), so two devices editing offline can clobber each other (last-write-wins). Decide
  merge vs. last-write before heavy multi-device use.
- **CSP not validated against a live deploy.** The CSP must allow the AI provider/worker origins (it
  whitelists the worker hostname) *and* Firebase Auth/Firestore; an untested CSP can silently break AI or
  auth traffic. KaTeX injects inline styles, complicating any move off `style-src 'unsafe-inline'`.
- **COPPA / age gating unresolved** (PRD open question) — now more pressing because minors type free text
  that feeds an LLM. Settle before real account collection.
- **Bundle/asset growth.** First-load is already heavy (§6.1); the 424-image scenery set (served on
  demand) and KaTeX fonts add to total weight; content is still bundled (good for now) but the PRD
  anticipates fetch-and-cache as the library grows.
- **Narrative coherence over long runs.** The `narrativeSummary` bounds prompt growth, but very long
  endless runs and the LLM's drift remain a quality risk to monitor.
- **`endStory` not wired to UI.** The reducer/controller support ending an adventure, but no screen
  exposes it (minor incomplete surface).
- **Local password is a demo boundary, not real security** (salted hash in `localStorage`; legacy
  accounts migrate to default `123456`).
- **Supply-chain weight.** `firebase-tools` + `openai` + `@google/genai` are large, fast-moving
  dependencies now in the lockfile (§6.11).
