coherent path — **Algebra Foundations** — through interactive, typed lesson steps with instant,

## 2. What the Project Is (Context)

- **Subject:** Algebra only — a deliberate depth-over-breadth bet (PRD "Non-Goals").
- **Persona:** "Maya," a new-to-algebra 8th grader, mobile-first.
- **Thesis:** Every lesson is a short sequence of *typed, interactive steps* rendered from a data
  model, with sub-100 ms client-side answer checking and specific authored feedback for both right
  and common wrong answers — never a bare red ✗.
- **Course path — a branching DAG** (not a straight line): Balancing Equations → One-Step →
  Two-Step, then Two-Step **diverges** into both **Like Terms** and **Coordinate Plane**, which
  **merge** as prerequisites of **Graphing Lines**. `prerequisites` on each lesson encode the graph;
  `lessonOrder` keeps a valid topological order for recommendations.

**Content model (`content/types.ts`):** A `Lesson` is an ordered array of tagged-union `LessonStep`s.
Step types number **ten**: `concept`, `mcq`, `input`, `operation-choice`, `sequence`, `balance`,
plus four hands-on types — **`manipulative`** (equal-groups / collect / build-product puzzles),
**`plot`** (interactive coordinate grid), **`slider`** (slope/intercept → live line), and
**`dragTerms`** (term-tile sorting). The engine exposes **eight** pure `check*Step` functions; of the
nine *assessed* step types, only **MCQ lacks an engine checker** (it is checked in the view — §6.6);
`concept` needs none. Each lesson file is 188–327 lines, fully authored — **none are "shells"**:

- **Balancing Equations** (266), **One-Step** (272), **Two-Step** (327), **Like Terms & Variables Both
  Sides** (309), **Coordinate Plane** (188), **Graphing Lines** (234).

**Equation rendering:** `equationToLatex` converts the app's plain authoring notation (e.g.
`"3x - 5 = 19"`, `"x / 6 = 2"`, `"x - 5 = 9 -> x = 9"`) into LaTeX with conservative, fail-soft
transforms (multi-letter prose → `\text{}`, `->` → `\rightarrow`, `a/b` → `\frac{a}{b}`, `*` →
`\cdot`). `MathText` renders it via `katex.render` into a ref (no `dangerouslySetInnerHTML`), with
`throwOnError:false`, an `aria-hidden` visual layer, and a visually-hidden plain-text label for
screen readers. KaTeX CSS is imported once in `main.tsx`.

**Design strengths:**

- **Clean separation of concerns.** Content (`content/`), pure logic (`engine/`), and persistence
  (`backend/` / `firebaseBackend.ts`) are decoupled; engine functions are pure and well tested.
  Lessons are one file each, which supports parallel authoring.
- **Provider-neutral backend behind one contract.** React talks only to `Backend`; both `LocalBackend`
  and `FirebaseBackend` implement it. The contract is async-capable (`MaybePromise`), and the UI
  loads learner data into React state and handles loading/error states (no reads during render).
- **Fails closed.** Selecting `firebase` without working config/SDK throws a clear startup error and
  refuses to silently fall back to local. The adapter glue is dynamically imported (`firebaseBackend`
  ~4 kB, `firebaseServices` ~0.26 kB), though the SDK itself currently rides the eager vendor chunk.
- **Security posture.** `firestore.rules` enforce per-user ownership, require the stored `userId`/`id`
  to match the path, gate learning-data writes on `email_verified`, keep attempts append-only, keep
  content read-only, and apply a default-deny catch-all. Firebase Auth adds email verification
  (send/resend/reload) enforced both client-side and in rules. `firebase.json` ships baseline security
  headers (CSP, HSTS, nosniff, Referrer-Policy, X-Frame-Options, Permissions-Policy).
- **Defensive persistence.** Both backends validate/normalize records; the local backend recovers from
  corrupt JSON, scopes access to the active session user, caches parsed reads, and never persists
  plaintext passwords. Firebase mastery/attempts writes use Firestore transactions (concurrency-safe;
  attempts reject overwrite).
- **CI & supply-chain hygiene.** Every push to `main`/`local` and every PR runs lint + build + test; a
  separate workflow runs `npm audit` (prod high/critical fails the build; dev findings report-only) and
  CodeQL (`security-extended`), plus a weekly schedule. Dependabot groups weekly npm and Actions
  updates.

## 6. Current Issues (concrete, present today)

### 6.1 Firebase SDK is not code-split out of the first load — also bloated by KaTeX (build regression)

`vite.config.ts` uses `manualChunks(id) => id.includes('node_modules') ? 'vendor' : undefined`. The
intent (cache the React runtime separately) is reasonable, but it sweeps **every dependency into one
`vendor` chunk** — React, the Firebase Auth/Firestore SDK, **and KaTeX**. Because React (and KaTeX,
used on the first lesson screens) are static dependencies of the entry, `vendor` is
**`modulepreload`ed and loaded eagerly** (confirmed in `dist/index.html`). Net effect: **every
visitor, including default local-mode users, downloads the ~1,015 kB / 304 kB-gzip vendor chunk that
still contains the unused Firebase SDK** on first load. The dynamic `import()` of the adapter only
splits ~4 kB of glue, not the heavy SDK it was meant to defer. This contradicts the README's
"Firebase is lazy-loaded" intent and pushes first-load JS to ~350 kB gzip, working against N1.
**Fix:** give Firebase its own manual chunk (so the SDK only loads in firebase mode), and optionally
split KaTeX into its own chunk for caching even though it is legitimately needed early. *(Note: a prior
report claimed a PR fixed this; the fix is **not present** in the current `main` — `vite.config.ts`
still uses the single-vendor approach.)*

### 6.2 KaTeX adds a meaningful first-render payload

Beyond the JS, KaTeX brings a **28.8 kB (7.9 kB gzip) stylesheet** and a **large font set** (the build
emits **59** font files across `.woff2`/`.woff`/`.ttf`). Browsers fetch only the glyph formats they
need, and only when math renders, but the first math-heavy screen can incur a brief font fetch / flash
of unstyled math. This is an acceptable trade for correct notation, but it should be measured against
N1 once deployed (and `font-display` / preloading the primary `.woff2` faces is worth considering). The
hosting CSP already allows `font-src 'self' data:`, which covers the self-hosted fonts.

### 6.3 Documentation drift — now partly contradicted by the code

The local-password feature (commit `225182f`) outran the docs, so several statements are now **wrong**,
not merely stale:

- **`SECURITY.md`** says "the default `LocalBackend` does not collect, store, or verify passwords …
  Local profile resume is email-only," and "No hosted backend yet. Firebase and Supabase adapters are
  not wired into the runtime." **Both are false now**: local mode verifies a salted-hash password, and
  `FirebaseBackend` is implemented and env-selectable. It also asserts "`npm audit` is expected to
  report zero vulnerabilities," which no longer holds for dev deps (§6.11).
- **`README.md`** still describes "the first three authored lessons, with shell lessons reserving the
  rest," lists "Next Priorities: Fill the Lesson 4-6 shells," and its smoke test references a
  "Lesson 4 shell." All six lessons are fully authored. It also does not mention KaTeX.
- **`BACKEND_ADAPTERS.md`** lists `mastery` as a "per-skill EWMA score" (it is a cumulative ratio —
  §6.5) and still points to `AuthScreen in src/App.tsx` (it now lives in `src/auth/AuthScreen.tsx`).

### 6.4 Equation-rendering code ships without tests

`equationToLatex` (a pure string→string transform with several regex branches) and `MathText` have
**no tests**. The converter is precisely the kind of pure function the repo already unit-tests
(cf. `authValidation`, `passwordCredential`), and its conservative regex transforms (prose-word
wrapping, fraction detection, arrow/`\cdot` substitution) have edge cases worth pinning (e.g. `x/6`
vs. a stray `/`, multi-letter variables, empty input). Add `tests/equation-latex.test.ts` to lock in
behavior before lessons author more notation against it.

### 6.5 Mastery model: cumulative ratio, not EWMA; coarse skill attribution

Both `LocalBackend` and `FirebaseBackend` store `score = Math.round((correct / attempts) * 100) / 100`
(no recency weighting/decay), so early misses count forever and the model can't support the Phase-3
decay/spaced-repetition plan unchanged. Separately, the app layer (`app/LearningApp.tsx`,
`completeStep`) updates mastery for **every** `skillId` of the active lesson with the same signal
(`activeLesson.skillIds.map(... updateSkillMastery ...)`), so multi-skill lessons (e.g. Balancing
Equations → `equality` + `inverse-operations`) always show identical per-skill scores.

### 6.6 MCQ logic lives in the view and duplicates engine escalation

Unlike the eight step types with engine checkers, there is no `checkMcqStep`; correctness
(`option.id === correctId`) and the hint→explanation→reveal escalation are reimplemented inline in
`MultipleChoiceStep`. A code comment even notes it "mirrors the engine's choice-step escalation," so
the two can drift apart.

### 6.7 No React error boundary; no runtime monitoring

Async flows set a user-visible `runtimeError` banner (good), but a render-time exception still has no
error boundary to contain it (and `MathText` adds a render-time third-party path). Runtime monitoring
(Crashlytics / Firebase Performance, per the PRD) is also not wired, so production errors and
N1/N2/N3 latency would be invisible after launch.

### 6.8 Password reset (R5) not implemented

Neither backend exposes a reset path. Firebase makes this near-free (`sendPasswordResetEmail`), and
the `AuthRepository` contract would need a matching method plus a small UI affordance. Without it, a
learner who forgets a password has no recovery in either mode.

### 6.9 Split UI/interaction modules remain untested

The former ~4,000-line `App.tsx` monolith has been broken into ~80 focused files under `app/`,
`lesson/`, `course/`, `screens/`, `auth/`, `components/`, and `hooks/` — a real maintainability win.
The remaining gap is coverage: these interaction/UI modules (drag/plot/slider/dragTerms renderers,
`MathText`) have **no automated tests** (the suite covers pure logic and content only), so a regression
in the split renderers would pass CI undetected until manual QA.

### 6.10 Toolchain Node-version sensitivity

The repo pins Node 24.16.0, and `scripts/node24-keepalive-fix.cjs` documents a **Node 24.17.0**
regression that breaks `firebase deploy` (it disables keep-alive as a workaround). Anyone deploying
must use the wrapper, and the pinned/CI Node should be kept off the broken patch.

### 6.11 New moderate dev-only npm advisories from `firebase-tools`

Committing `firebase-tools` as a devDependency (for fast local deploys) added **5 moderate** advisories
via `@opentelemetry/core` (unbounded memory in W3C Baggage), `@google-cloud/pubsub`, `uuid` (missing
buffer bounds check), and `gaxios`. These are **dev-only** — `npm audit --omit=dev` is clean and the CI
production gate (`--audit-level=high`) still passes — but they contradict `SECURITY.md`'s "zero
vulnerabilities" claim. The clean fix (`npm audit fix --force`) would downgrade `firebase-tools` to
14.x (a breaking change), so the pragmatic move is to track the advisories, wait for an upstream patch,
and update `SECURITY.md` to scope the "zero" expectation to production deps.

## 8. Potential / Future Issues (risks to watch)

- **Cross-device sync/conflict semantics.** Firestore writes overwrite whole `LessonProgress` /
  `SkillMastery` documents, so two devices editing offline can clobber each other (last-write-wins).
  Decide merge vs. last-write semantics before heavy multi-device use (the BrainLift flags the same
  question about Brilliant itself).
- **CSP not validated against a live deploy.** The README explicitly TODOs verifying the CSP, replacing
  `style-src 'unsafe-inline'`, and HSTS `preload` after a real deploy; an untested CSP can silently
  break Auth/Firestore traffic or provide false confidence. (KaTeX injects inline styles, which
  interacts with any move away from `'unsafe-inline'`.)
- **COPPA / age gating unresolved** (PRD open question) for a teen audience — must be settled before any
  real account collection.
- **Bundle/content growth.** With the SDK still eager and KaTeX added (§6.1/§6.2), first-load is already
  heavy; content is bundled (good for now) but the PRD anticipates fetch-and-cache as the library grows.
- **Supply-chain drift from `firebase-tools`.** A large, fast-moving CLI now lives in the lockfile and
  is the source of the only outstanding advisories (§6.11); keep it patched, or consider invoking it
  via `npx` on demand rather than pinning it as a devDependency.
- **Local password is a demo boundary, not real security.** The salted hash lives in `localStorage`
  and legacy accounts migrate to the default password `123456`; this is fine for a single-device demo
  but must not be mistaken for production auth.
- **Mobile drag uses point hit-testing.** The keyboard/tap fallback mitigates accessibility, but the
  pointer-drag path still warrants real iOS/Android Safari/Chrome testing.
- **No interaction-layer tests.** CI guards the pure logic, but a regression in the UI/interaction
  modules or in `MathText` rendering would pass CI undetected until manual QA (§6.9).
